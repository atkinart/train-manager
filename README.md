# train-manager

# BRIO Automation — архитектура и план реализации

## 1. Цель проекта

Построить расширяемую систему автоматизации для BRIO-железной дороги с 3D-печатными элементами, которая умеет:

* определять, **какой поезд** прошёл через конкретную точку;
* переключать **стрелки** по правилам;
* в будущем управлять **шлагбаумами**, светофорами и другими исполнительными механизмами;
* передавать события через **MQTT/Mosquitto**;
* отображать текущее состояние макета в **web UI**;
* быть понятной и расширяемой при небольшом опыте с Arduino, MQTT и автоматикой.

---

## 2. Принятые технические решения

### Уже выбранный стек

* **RFID-считыватели:** MFRC522
* **Контроллеры на макете:** Arduino
* **Брокер сообщений:** MQTT / Mosquitto
* **Backend:** Java + Spring Boot
* **Frontend:** JavaScript Web UI
* **Исполнительные механизмы стрелок:** сервоприводы

### Почему такая архитектура

Разделяем систему на 3 слоя:

1. **Полевой уровень (Arduino)**

   * считывает RFID;
   * управляет стрелками и другими механизмами;
   * публикует телеметрию;
   * подписывается на команды.

2. **Логический уровень (Spring Boot)**

   * хранит модель макета;
   * принимает события от Arduino;
   * применяет правила;
   * публикует команды на Arduino;
   * отдаёт данные в web UI.

3. **Визуализация (JS Web UI)**

   * карта путей;
   * положение стрелок;
   * журнал событий;
   * статус поездов и участков.

Это позволяет не перегружать Arduino сложной логикой.

## 3. Архитектура системы

## 3.1 Логическая схема

```text
[Поезд + RFID tag]
        ↓
[MFRC522 reader на макете]
        ↓ SPI
[Arduino node]
        ↓ MQTT publish/subscribe
[Mosquitto broker на Raspberry Pi]
        ↓
[Spring Boot backend]
   ├─ Rule Engine
   ├─ Layout State Store
   ├─ REST / WebSocket API
   └─ Event Log
        ↓
[Web UI]
```

## 3.2 Роли компонентов

### Поезд

* несёт RFID-метку;
* метка уникально идентифицирует поезд.

### MFRC522

* установлен в конкретной точке макета;
* считывает UID тега, когда поезд проходит над ним.

### Arduino node

* привязан к участку макета;
* знает свои RFID-считыватели и сервоприводы;
* отправляет события в MQTT;
* получает команды на переключение стрелок.

### Mosquitto

* центральная шина сообщений;
* развязывает Arduino и backend.

### Spring Boot

* единый источник логики и состояния;
* преобразует события в действия;
* хранит карту макета, поезда, стрелки, правила.

### Web UI

* показывает текущее состояние;
* даёт ручное управление;
* показывает лог событий.

## 4. Модель событий

Система строится вокруг событий.

### Базовые события

* `train detected at marker`
* `switch changed`
* `reader online/offline`
* `arduino node online/offline`
* `command accepted/rejected`

### Пример бизнес-события

* поезд `train_3` прошёл метку `marker_station_in`
* backend определяет правило
* backend публикует команду `switch main_1 -> LEFT`
* Arduino исполняет команду
* Arduino публикует подтверждение

## 5. MQTT-дизайн

## 5.1 Общий принцип

* Arduino публикует события в `events/...`
* Spring публикует команды в `commands/...`
* Arduino публикует подтверждения в `state/...`

## 5.2 Рекомендуемые топики

### События от Arduino

```text
events/node/{nodeId}/rfid/{readerId}
events/node/{nodeId}/switch/{switchId}
events/node/{nodeId}/status
events/node/{nodeId}/power
```

### Команды от backend

```text
commands/node/{nodeId}/switch/{switchId}/set
commands/node/{nodeId}/ping
commands/node/{nodeId}/config
```

### Состояние устройств

```text
state/node/{nodeId}/switch/{switchId}
state/node/{nodeId}/reader/{readerId}
state/node/{nodeId}/health
```

## 5.3 Формат сообщений JSON

### RFID event

```json
{
  "eventId": "8d4f1b9e-4a8f-4d4c-8a36-58fb61234abc",
  "nodeId": "yard-node-1",
  "readerId": "reader-A1",
  "tagUid": "04A1B2C3D4",
  "timestamp": "2026-03-27T12:00:00Z",
  "rssi": null
}
```

### Команда на стрелку

```json
{
  "commandId": "cmd-00021",
  "switchId": "switch-1",
  "position": "LEFT",
  "issuedAt": "2026-03-27T12:00:02Z",
  "source": "rule-engine"
}
```

### Подтверждение состояния стрелки

```json
{
  "switchId": "switch-1",
  "position": "LEFT",
  "status": "DONE",
  "timestamp": "2026-03-27T12:00:02Z"
}
```

## 6. Архитектура по железу

## 6.1 Принцип разбиения по узлам

Не нужно тянуть все датчики и все сервы к одной Arduino.

Лучше делать **узлы макета**:

* `Node A` — станция
* `Node B` — развилка
* `Node C` — переезд

Каждый узел содержит:

* 1 Arduino
* 1..N RFID reader
* 1..N servos
* питание
* MQTT-связь через центральный уровень

### Рекомендуемый старт

Для первой версии:

* 1 Arduino node
* 2 MFRC522
* 2 servo для стрелок
* 1 Raspberry Pi с Mosquitto + Spring + UI

## 8. Питание — самое важное

Это критическая часть. Главная ошибка новичков: питать сервы прямо от 5V Arduino.

### Нельзя

* питать несколько сервоприводов от 5V пина Arduino;
* вешать сервы и RFID без общей земли;
* питать всё одним слабым USB без проверки тока.

### Нужно

* **сервоприводы питаются от отдельного 5V БП**;
* **Arduino питается отдельно** (USB или свой стабилизированный канал);
* **земля у всех общая**: GND Arduino, GND сервоприводов, GND RFID, GND внешнего 5V блока должны быть соединены.

## 8.1 Рекомендуемая схема питания для старта

```text
220V
 └─ 5V power supply (3A или больше)
      ├─ Servo #1 V+
      ├─ Servo #2 V+
      ├─ Arduino 5V input / USB power
      └─ MFRC522 modules via regulated rail

All GND connected together
```

## 8.2 Практическая рекомендация

Для первой версии:

* Raspberry Pi питается своим штатным БП;
* Arduino можно питать по USB;
* сервы питаются от отдельного 5V БП;
* земля Arduino и серво-БП объединяется.

Так меньше риск нестабильности.

## 8.3 Если позже появится общая силовая шина

Можно сделать:

* 12V общая шина по макету;
* в каждом узле DC-DC преобразователь 12V -> 5V;
* этим 5V питать Arduino и сервы локально.

Это более взрослая архитектура.

## 9. Подключение MFRC522

## 9.1 Важное замечание

MFRC522 — **3.3V устройство**.

Это значит:

* питание модуля: 3.3V;
* нужно внимательно смотреть конкретный модуль;
* нельзя бездумно подавать на него 5V логические уровни, если плата не рассчитана на это.

Для первого прототипа лучше:

* использовать один проверенный тип модулей;
* собрать сначала 1 reader;
* убедиться в стабильности чтения.

## 9.2 Подключение по SPI

Типовые линии:

* SDA/SS
* SCK
* MOSI
* MISO
* RST
* 3.3V
* GND

### Пример логического подключения к Arduino

```text
MFRC522 #1 -> Arduino
SCK  -> SPI SCK
MOSI -> SPI MOSI
MISO -> SPI MISO
SS   -> отдельный digital pin
RST  -> отдельный digital pin
3.3V -> 3.3V
GND  -> GND
```

Если считывателей несколько:

* SCK/MOSI/MISO общие;
* у каждого reader свой `SS`;
* можно использовать общий `RST` или отдельные линии, если нужно.

## 9.3 Что важно в механическом монтаже

* метка должна проходить **максимально близко** к антенне;
* между меткой и reader не должно быть лишнего расстояния;
* лучше закладывать место в 3D-модели под точную юстировку;
* сначала нужно протестировать реально достижимую дистанцию чтения.

## 10. Подключение стрелок

## 10.1 Стартовый вариант — напрямую к Arduino

Для 2 стрелок можно сделать так:

* Servo #1 signal -> digital pin
* Servo #2 signal -> digital pin
* питание сервоприводов отдельно 5V
* GND общий

## 10.2 Масштабируемый вариант — через PCA9685

Когда стрелок станет больше:

* Arduino управляет PCA9685 по I2C;
* PCA9685 даёт много каналов PWM;
* сервы питаются от отдельной силовой линии;
* Arduino почти не загружен генерацией PWM.

### Когда переходить на PCA9685

Сразу не обязательно.

Но стоит закладывать в архитектуру, если:

* стрелок будет больше 4;
* появятся шлагбаумы;
* хочется стандартизировать управление сервами.

## 11. Рекомендуемая структура узла Arduino

## 11.1 Задачи Arduino

Arduino должен быть максимально простым:

* инициализировать RFID readers;
* периодически опрашивать readers;
* фильтровать дребезг / повторные чтения;
* управлять сервоприводами;
* обмениваться сообщениями с верхним уровнем;
* публиковать heartbeat.

## 11.2 Что не надо класть в Arduino

Не надо хранить на Arduino:

* сложную маршрутизацию;
* глобальную модель макета;
* правила уровня всей станции;
* web UI логику.

Это должен делать backend.

## 12. Структура backend (Spring Boot)

## 12.1 Модули backend

### 1. MQTT Gateway

* подписка на `events/#` и `state/#`;
* публикация в `commands/#`.

### 2. Device Registry

* список узлов;
* readers;
* switches;
* соответствие `RFID UID -> trainId`.

### 3. Layout Model

* карта макета;
* блок-участки;
* связи между участками и стрелками.

### 4. Rule Engine

* получает события;
* вычисляет нужные переключения;
* публикует команды.

### 5. Event Log

* хранение истории событий.

### 6. REST/WebSocket API

* отдаёт текущее состояние UI;
* пушит обновления в реальном времени.

## 12.2 Правила переключения

Рекомендуется хранить их декларативно.

### Пример

```json
{
  "ruleId": "route_station_entry",
  "when": {
    "readerId": "reader-A1",
    "trainId": "train-red"
  },
  "then": [
    { "switchId": "switch-1", "position": "LEFT" },
    { "switchId": "switch-2", "position": "RIGHT" }
  ]
}
```

### Следующий уровень

Позже можно добавлять условия:

* занятость участка;
* текущее положение стрелки;
* направление движения;
* расписание / сценарий.

## 15. Практика по Arduino — что именно учить

Так как опыта немного, двигаться лучше маленькими изолированными шагами.

## 15.1 Порядок обучения

1. **Digital I/O**

   * понять пины, `HIGH/LOW`, `INPUT/OUTPUT`

2. **Servo**

   * подключить 1 серво;
   * выставлять 2 фиксированных положения;
   * понять питание и общую землю.

3. **SPI**

   * понять общие линии и отдельный `SS`;
   * подключить 1 MFRC522.

4. **JSON / Serial debug**

   * печатать события в Serial;
   * отлаживать структуру сообщений.

5. **MQTT transport**

   * сначала через центральный уровень;
   * не усложнять на старте.

## 15.2 Правило отладки

Никогда не отлаживать сразу всё вместе.

Правильный порядок:

* отдельно reader;
* отдельно servo;
* отдельно MQTT;
* потом объединение.

## 17. Риски и подводные камни

## 17.1 RFID

* маленькая дистанция чтения;
* чувствительность к геометрии;
* не всегда гарантированное чтение на скорости.

### Что делать

* тестировать реальную геометрию на столе;
* не встраивать сразу в финальную печать;
* дать возможность регулировки положения reader.

## 17.2 Сервы

* просадка питания;
* дрожание;
* механический люфт.

### Что делать

* отдельное питание;
* ограничители хода;
* настройка конечных углов.

## 17.3 Arduino

* нехватка памяти при усложнении;
* сложность большого количества устройств на одном узле.

### Что делать

* не перегружать один узел;
* переносить логику в backend.

## 17.4 MQTT/Backend

* гонки между командами;
* рассинхронизация состояния.

### Что делать

* всегда публиковать подтверждение исполнения;
* иметь единый источник правды в backend.

## 21. Следующий документ, который нужно сделать после этого

После утверждения архитектуры логично сделать ещё 3 документа:

1. **BOM / shopping list**

   * конкретные позиции и количества

2. **Wiring guide**

   * какие пины куда подключать

3. **Software skeleton**

   * структура репозиториев и базовые классы
